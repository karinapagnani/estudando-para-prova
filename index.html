Hardware Essencial

⚡

CPU — Unidade Central de Processamento

HARDWARE

O "cérebro" do computador. Executa instruções de programas buscando-as na memória, decodificando e executando em ciclos contínuos (busca → decodifica → executa).

Componentes internos:
ULA — cálculos Registradores — armazenamento rápido Unidade de Controle — orquestra tudo Cache L1/L2/L3 Barramento interno
Ciclo de instrução: Fetch (busca na RAM) → Decode (interpreta opcode) → Execute (ULA ou memória) → Write-back (salva resultado)
Dica de prova: A velocidade da CPU é medida em GHz (bilhões de ciclos/seg). Mais núcleos = mais processos em paralelo.
🧮

ULA — Unidade Lógica e Aritmética

HARDWARE

Componente da CPU responsável por realizar todas as operações matemáticas (soma, subtração, etc.) e lógicas (AND, OR, NOT, comparações).

Tipo de operaçãoExemplos
Aritméticas+, -, ×, ÷, módulo
LógicasAND, OR, NOT, XOR
Comparação>, <, ==, !=
Deslocamento de bitsshift left, shift right
A ULA recebe operandos dos registradores e devolve o resultado para um registrador de destino. Ela não lê memória diretamente.
📌

Registradores

CPU

Memória ultra-rápida dentro da CPU. Armazenam temporariamente os dados que estão sendo processados agora mesmo. São os mais rápidos, mas os de menor capacidade.

RegistradorFunção
PC (Program Counter)Guarda o endereço da próxima instrução
IR (Instruction Register)Instrução atualmente em execução
SP (Stack Pointer)Topo da pilha de chamadas
ACC (Acumulador)Resultado temporário de operações
MAR / MDREndereço/dado a ser lido ou escrito na RAM
Hierarquia de velocidade: Registradores > Cache L1 > Cache L2 > RAM > HD/SSD
💾

RAM — Memória de Acesso Aleatório

MEMÓRIA

Memória principal do sistema. Volátil — perde tudo quando desligado. Armazena programas e dados em uso. A CPU acessa a RAM constantemente durante a execução.

Analogia: Se o HD é a estante de livros, a RAM é a mesa de trabalho. Você só consegue trabalhar com o que está na mesa. Mesa maior = mais programas abertos ao mesmo tempo.
RAM vs Cache vs HD:
RAM: rápida, volátil, cara por GB
Cache: rapidíssima, dentro da CPU, poucos MB
HD/SSD: lenta, persistente, barata, GBs ou TBs
🗄️

HD — Disco Rígido (Hard Disk)

ARMAZENAMENTO

Armazenamento persistente e não-volátil. Mantém dados mesmo sem energia. Muito mais lento que a RAM, mas capacidade enorme. Base para o sistema de arquivos e o Swap.

TipoTecnologiaVelocidade
HDDMecânico (discos giratórios)~100 MB/s
SSDFlash (chips)~500 MB/s+
NVMe SSDFlash + PCIe~3000 MB/s+
Papel no SO: Armazena o SO, programas instalados, arquivos do usuário, e também a partição de Swap (memória virtual).
🎮

Controladoras

E/S

Chips ou circuitos que gerenciam a comunicação entre a CPU e os dispositivos de entrada/saída (teclado, HD, GPU, USB, etc.). Atuam como intermediários inteligentes.

Como funciona: A CPU envia um comando à controladora, que gerencia o dispositivo de forma independente. Quando termina, a controladora gera uma interrupção avisando a CPU.
Controladora de disco (SATA/NVMe) Controladora USB Controladora de vídeo (GPU) Controladora de rede (NIC)
Sem controladoras, a CPU teria que gerenciar cada bit transferido para um HD — ficaria ocupada 100% do tempo só com I/O.

Sistemas Operacionais

🏗️

Máquina Multinível

ARQUITETURA

Computadores funcionam em camadas de abstração. Cada nível oferece uma "máquina virtual" para o nível acima, ocultando a complexidade do hardware.

Nível 5 — Linguagens de Alto Nível (C, Java, Python) Nível 4 — Linguagem Assembly Nível 3 — Sistema Operacional (chamadas de sistema) Nível 2 — ISA (Instruction Set Architecture) Nível 1 — Microarquitetura (lógica de controle) Nível 0 — Lógica Digital (portas lógicas, transistores)
Cada nível só precisa "falar" com o nível imediatamente abaixo. Um programador Python não precisa saber de transistores!
📋

JCL — Job Control Language

HISTÓRICO

Linguagem de controle de tarefas usada em mainframes IBM. Usada para descrever jobs (trabalhos em lote) — diz ao SO quais programas executar, quais arquivos usar e quanta memória alocar.

Contexto histórico: Era dos sistemas batch (lote). Programadores entregavam cartões perfurados com JCL ao operador, que rodava os jobs durante a madrugada. Resultado vinha horas depois. 
// Exemplo JCL simplificado
//MYJOB   JOB  (123),'MEU NOME'
//STEP1   EXEC PGM=MYPROG
//INPUT   DD   DSN=MEU.ARQUIVO,DISP=SHR
//OUTPUT  DD   SYSOUT=*
Na prova: JCL representa a era dos sistemas monotarefa/batch. Um job de cada vez, sem interação do usuário.
🖥️

GUI — Interface Gráfica

INTERFACE

Interface que usa elementos visuais (janelas, ícones, botões, menus) em vez de texto puro. Torna o computador acessível para qualquer pessoa — não só programadores.

CLI (Terminal)GUI (Interface Gráfica)

Conceito WIMP: Windows (janelas), Icons (ícones), Menus, Pointer (ponteiro) — os 4 elementos fundamentais de qualquer GUI.
⚙️

Chamadas de Sistema (System Calls)

SO

A interface entre programas e o SO. Quando um programa precisa usar hardware (ler arquivo, criar processo, alocar memória), ele faz uma chamada de sistema — pedindo permissão ao SO.

Programa (modo usuário) │ │ system call (ex: read(), fork(), malloc()) ▼ [ Kernel — modo privilegiado ] │ ▼ Hardware (disco, CPU, RAM...)
open() / close() read() / write() fork() — cria processo exec() — executa programa exit() — encerra processo malloc() — aloca memória
Por que isso existe? Sem system calls, qualquer programa malicioso poderia acessar diretamente o HD, outros processos, a rede... Caos total!
🔀

Monotarefa vs Multitarefa

TIPOS DE SO

Define quantos processos o SO executa simultaneamente. Monotarefa: um de cada vez. Multitarefa: vários processos "ao mesmo tempo" (na prática, compartilham a CPU).

MonotarefaMultitarefa
MS-DOS, CP/MWindows, Linux, macOS
1 processo ativo por vezN processos simultâneos
Simples, previsívelComplexo, requer escalonador
CPU ociosa durante I/OCPU ocupa outro processo enquanto aguarda I/O
Como funciona a multitarefa real? A CPU alterna entre processos muito rapidamente (milissegundos). Parece simultâneo, mas é concorrente. Com múltiplos núcleos, é de fato paralelo.
🔬

Microkernel

ARQUITETURA SO

Filosofia de design de SO onde o kernel é mínimo — só as funções essenciais ficam no núcleo. Todo o resto (drivers, sistema de arquivos, rede) roda em modo usuário como servidores separados.

Monolítico (ex: Linux)Microkernel (ex: Minix, QNX)
Tudo no kernelKernel mínimo (IPC, scheduler, memória básica)
Mais rápido (sem overhead de IPC)Mais seguro e estável
Bug no driver = kernel crashBug no driver ≠ crash do sistema
Difícil de modificar partesFácil trocar componentes
Exemplo prático: Se o driver de áudio travar em um microkernel, só o serviço de áudio cai — o SO continua funcionando. No monolítico, pode travar tudo.
🧵

Threads

CONCORRÊNCIA

Unidade básica de execução dentro de um processo. Um processo pode ter várias threads que compartilham a mesma memória, mas executam fluxos de código independentes.

Processo (espaço de memória compartilhado) ├── Thread 1 → executa função A() ├── Thread 2 → executa função B() └── Thread 3 → aguarda I/O (todas compartilham heap, código, globals)
ProcessoThread
Memória independenteCompartilha memória do processo
Criação lenta (fork)Criação rápida
Isolado — falha não propagaFalha pode matar todas as threads
IPC necessário para comunicarComunicação direta (mesma memória)
Aplicação prática: Servidor web cria uma thread por requisição. Navegador usa thread para UI, outra para rede, outra para renderização.
03

Gerenciamento de Memória

🗺️

MMU — Memory Management Unit

HARDWARE+SO

Componente de hardware que traduz endereços virtuais em físicos. Permite que cada processo "veja" sua própria memória isolada, mesmo que fisicamente esteja espalhada na RAM.

Processo A pensa que está no endereço 0x1000 │ ▼ [ MMU ] ← tabela de páginas │ ▼ RAM física: endereço 0x4F200
Benefícios da MMU:
• Isolamento entre processos (segurança)
• Memória virtual maior que a RAM física
• Permite o Swap
• Base da proteção de memória do SO
Sem MMU: Qualquer processo poderia ler/escrever a memória de outro. Era o caso dos primeiros PCs com MS-DOS.
🔄

Swap (Swap In / Swap Out)

MEMÓRIA VIRTUAL

Mecanismo que usa espaço em disco como extensão da RAM. Quando a RAM enche, páginas inativas são movidas ao disco (Swap Out). Quando necessárias, voltam (Swap In).

MAPA DE MEMÓRIA
0x0000
Kernel / SO
0x1000
Processo A (ativo)
0x2000
Processo B (ativo)
0x3000
⟵ SWAP OUT (foi pro disco)
0x4000
Livre
Swap Out: Página da RAM → Disco (quando RAM cheia)
Swap In: Página do Disco → RAM (quando processo precisa dela)

⚠️ Cuidado: Swap é muito mais lento que RAM. Excesso de swap = "thrashing" — o sistema fica lento demais.
04

Escalonamento de Processos

📥

FIFO — First In, First Out

ESCALONAMENTO

O processo que chega primeiro é executado primeiro. Simples e justo na chegada, mas pode causar o "efeito comboio" — um processo longo trava todos os demais.

FILA DE PROCESSOS (ordem de chegada):
P110ms
P22ms
P35ms
P41ms
→ Executa P1 (10ms) → P2 (2ms) → P3 (5ms) → P4 (1ms)
Tempo médio de espera: P1=0, P2=10, P3=12, P4=17 → média = 9.75ms

Problema: P4 (1ms) espera 17ms por culpa do P1 (10ms). Isso é o "efeito comboio".
⚡

SJF — Shortest Job First

ESCALONAMENTO

Executa o processo mais curto primeiro. Minimiza o tempo médio de espera — é o algoritmo ótimo nesse critério. Porém, pode causar inanição (starvation) de processos longos.

MESMA FILA, REORDENADA POR TAMANHO:
P41ms
P22ms
P35ms
P110ms
→ Executa P4 (1ms) → P2 (2ms) → P3 (5ms) → P1 (10ms)
Tempo médio de espera: P4=0, P2=1, P3=3, P1=8 → média = 3ms

Muito melhor que FIFO (9.75ms)! Mas o problema: como o SO sabe quanto tempo um processo vai demorar? Geralmente usa estimativas baseadas no histórico.
Starvation: Se sempre chegam processos curtos, o processo longo nunca executa. Solução: Aging (aumentar prioridade com o tempo de espera).
Comparativo rápido de escalonadores:

FIFO/FCFS — simples, não-preemptivo, justo na chegada, sujeito a efeito comboio
SJF — ótimo em tempo médio de espera, não-preemptivo, pode causar starvation
Round Robin — preemptivo, usa time slice, justo para todos, bom para interatividade
Prioridade — executa por nível de prioridade, pode causar starvation (resolvido com aging)

DHCP, DNS e MAC

PROTOCOLOS

Três conceitos fundamentais de redes que aparecem muito em provas de S.O. e Redes.

DHCP (Dynamic Host Configuration Protocol): distribui endereços IP automaticamente para dispositivos na rede. Sem ele, cada IP seria configurado manualmente.
DNS (Domain Name System): traduz nomes (google.com) em IPs (142.250.x.x). É a "agenda telefônica" da internet.
MAC (Media Access Control): endereço físico único de 48 bits gravado na placa de rede. Usado na camada 2 (Enlace). Switches encaminham com base no MAC.

Modelo OSI — Detalhado

O que é o Modelo OSI?

O OSI (Open Systems Interconnection) é um modelo de referência criado pela ISO em 1984 para padronizar a comunicação entre sistemas de diferentes fabricantes. Ele divide o processo de comunicação em 7 camadas independentes, onde cada camada oferece serviços para a camada acima e usa os serviços da camada abaixo.

O modelo OSI não é um protocolo em si — é um framework conceitual. Os protocolos reais (TCP/IP, HTTP, Ethernet) são implementações que se encaixam nesse modelo. Na prática, a internet usa o modelo TCP/IP, que tem 4 camadas, mas o OSI é o modelo didático padrão do mundo acadêmico.

MNEMÔNICO

4 — Transporte

"Através Para Sessões,
Transportamos Redes Em Fragmentos"
Encapsulamento: Quando dados descem pelas camadas (no emissor), cada camada adiciona um cabeçalho (header) com suas informações de controle. Quando sobem (no receptor), cada camada remove seu cabeçalho e entrega para a de cima. Esse processo chama-se encapsulamento/desencapsulamento.

É a camada mais próxima do usuário. Não é o aplicativo em si (como o Chrome ou o Outlook), mas a interface entre o software e a rede. Define como os programas se comunicam — os protocolos que determinam o formato e a semântica das mensagens trocadas.

HTTP vs HTTPS: O HTTP trafega dados em texto puro — qualquer intermediário pode ler. O HTTPS encapsula o HTTP dentro do TLS, criando um canal criptografado. Isso é feito na camada de Apresentação (6), mas iniciado aqui na camada 7.

2. Seu SO consulta o cache local
3. Se não tiver, consulta o servidor DNS
4. DNS devolve o IP: 142.250.x.x
5. Seu navegador abre conexão com esse IP

6

Camada de Apresentação

PDU: Dados (Data)  |  Tradução, Criptografia, Compressão
PRESENTATION

Responsável pela sintaxe e semântica dos dados — garante que a informação enviada por um sistema seja compreendida pelo outro, mesmo que usem formatos internos diferentes. É aqui que acontecem criptografia, compressão e conversão de formatos.

TRÊS FUNÇÕES PRINCIPAIS
1. Tradução/Conversão de formato
Converte dados entre formatos diferentes. Ex: ASCII ↔ EBCDIC, little-endian ↔ big-endian, JSON ↔ XML. Garante que os dados cheguem no formato que a aplicação espera.
2. Criptografia / Descriptografia
Cifra os dados antes de enviar e decifra ao receber. O protocolo TLS/SSL opera aqui — é ele que dá o "S" do HTTPS. Usa algoritmos como AES, RSA, ChaCha20.
3. Compressão / Descompressão
Reduz o tamanho dos dados para economizar banda. Formatos como JPEG, MP3, ZIP, GZIP atuam nessa camada antes de os dados serem transmitidos.
EXEMPLOS PRÁTICOS
TecnologiaO que faz aqui
TLS/SSLCriptografa a conexão HTTPS
JPEG/PNGCompressão de imagens
MP3/AACCompressão de áudio
ZIP/GZIPCompressão de arquivos
ASCII/UTF-8Codificação de caracteres
Base64Codificação de binário em texto
Nota acadêmica: Na prática do modelo TCP/IP, a camada de Apresentação é absorvida pela camada de Aplicação. No OSI puro ela existe para separar claramente a responsabilidade de "tradução de dados" da "lógica de aplicação".
5

Camada de Sessão

PDU: Dados (Data)  |  Estabelece, gerencia e encerra sessões
SESSION

Gerencia o diálogo entre os processos nas duas extremidades — abre, mantém, sincroniza e encerra sessões de comunicação. Uma sessão é uma conexão lógica entre aplicações, que pode durar horas ou dias, independente das conexões TCP subjacentes.

RESPONSABILIDADES
Estabelecimento de sessão: Negocia e inicia a comunicação entre duas aplicações. Define parâmetros como autenticação inicial e permissões.
Manutenção e sincronização: Insere pontos de verificação (checkpoints) no fluxo de dados. Se a transmissão falhar, pode retomar de um checkpoint — não precisa recomeçar do zero. Essencial em transferências longas.
Encerramento de sessão: Garante que a sessão seja fechada corretamente (graceful close), liberando recursos. Sem isso, conexões ficariam abertas indefinidamente.
PROTOCOLOS E EXEMPLOS
TecnologiaUso na camada 5
NetBIOSSessões em redes Windows legado
RPCChamada remota de procedimentos
PPTPGerencia túneis VPN
SIPSessões de VoIP (ligações pela internet)
SQL SessionsSessões de banco de dados
Exemplo real: Quando você faz login em um banco online, a camada de sessão mantém sua sessão autenticada. Se a conexão cair e você reconectar, a sessão pode ser retomada sem novo login (dependendo da implementação).

Camada de Transporte

PDU: Segmento (TCP) / Datagrama (UDP)  |  Portas, confiabilidade, segmentação
TRANSPORT

Responsável pela comunicação ponto a ponto entre processos (identificados por portas), pela segmentação dos dados em pedaços menores e pela confiabilidade da entrega. É aqui que vivem os protocolos TCP e UDP.

Segmentação: A camada de Transporte divide dados grandes em segmentos menores para transmissão. Cada segmento tem um número de sequência. No destino, a camada de Transporte os reordena e remonta. Se um segmento se perder, só ele é retransmitido (no TCP).
Por que UDP é mais rápido? UDP não espera confirmações, não reordena, não controla fluxo. Em streaming de vídeo, é melhor perder um frame do que pausar esperando a retransmissão — por isso YouTube e Netflix usam protocolos sobre UDP (como QUIC).

Máscara:  255  . 255  . 255 .  0    (/24)
Rede:     192.168.1.0

Como o roteamento funciona: Cada roteador tem uma tabela de roteamento. Ao receber um pacote, ele lê o IP destino, consulta a tabela e encaminha pela interface correta — de salto em salto (hop by hop) até o destino.
IPv4 vs IPv6:
IPv4: ~4,3 bilhões de endereços (esgotados!)
IPv6: 2¹²⁸ ≈ 340 undecilhões de endereços
ARP — Address Resolution Protocol: Quando você quer enviar dados para 192.168.1.50, sua máquina sabe o IP mas não sabe o MAC (necessário para Ethernet). O ARP faz um broadcast: "quem tem o IP 192.168.1.50?". O dono responde com seu MAC. Esse mapeamento fica cacheado na tabela ARP local.

Camada de Enlace de Dados

PDU: Quadro (Frame)  |  Endereçamento físico (MAC), controle de acesso ao meio
DATA LINK

Responsável pela comunicação entre dois dispositivos diretamente conectados (mesmo segmento de rede). Usa endereços MAC (físicos) para identificar dispositivos na rede local. Detecta e corrige erros de transmissão física. O Switch opera aqui.

DUAS SUBCAMADAS IMPORTANTES
LLC — Logical Link Control:
Subcamada superior. Controla o fluxo e gerencia erros entre os frames. Identifica qual protocolo de camada 3 está sendo usado (IPv4, IPv6, ARP). Faz a interface com a camada de Rede.
MAC — Media Access Control:
Subcamada inferior. Controla o acesso ao meio físico compartilhado (quem pode transmitir quando). Define o endereço MAC: 48 bits em hex (ex: AA:BB:CC:DD:EE:FF). Os primeiros 24 bits identificam o fabricante (OUI).
ESTRUTURA DO FRAME ETHERNET
┌──────────┬──────────┬──┬──────────────┬───────┐ │ MAC DST │ MAC SRC │T │ DADOS │ FCS │ │ 6 bytes │ 6 bytes │2b│ 46-1500 bytes│4 bytes│ └──────────┴──────────┴──┴──────────────┴───────┘ MAC DST = endereço MAC destino MAC SRC = endereço MAC origem T = EtherType (0x0800=IPv4, 0x86DD=IPv6) FCS = Frame Check Sequence (detecção de erro)
Protocolos: Ethernet, Wi-Fi (802.11), PPP, HDLC, Frame Relay, ATM
CSMA/CD (Ethernet cabeada): Antes de transmitir, escuta o meio. Se colisão detectada, para e aguarda tempo aleatório antes de tentar de novo.
Switch vs Hub nessa camada: O Hub (camada 1) repete o frame para TODAS as portas — cria um domínio de colisão único. O Switch (camada 2) lê o MAC destino e encaminha o frame APENAS para a porta correta, criando domínios de colisão separados por porta. Resultado: muito mais eficiente e seguro.

Camada Física

PDU: Bit  |  Transmissão de bits pelo meio físico
PHYSICAL

A camada mais baixa. Responsável pela transmissão bruta de bits pelo meio físico — define voltagens, frequências, conectores, cabos, sinais elétricos, ópticos ou de rádio. Não entende frames, pacotes ou dados — só 0s e 1s.

CONCEITOS TÉCNICOS
Banda larga vs Banda base:
Banda base: sinal digital direto no cabo (Ethernet)
Banda larga: múltiplos canais por modulação (DSL, cabo)
Modulação: Técnica de codificar bits em sinal analógico. AM, FM, PSK, QAM. O modem (MOdulador/DEModulador) opera aqui.
Multiplexação:
FDM — divide por frequência
TDM — divide por time slot
WDM — divide por comprimento de onda (fibra)
Dispositivos da camada 1: Hub, Repetidor, Modem, Cabo de rede, Conectores RJ-45, Transceiver. Todos eles apenas amplificam ou repetem o sinal elétrico/óptico sem interpretar o conteúdo.

Cheat Sheet — Revisão Relâmpago

HARDWARE
CPU = busca + decodifica + executa
ULA = cálculos e lógica
Registradores = memória ultrarrápida na CPU
RAM = memória volátil e rápida
HD = armazenamento persistente
Controladora = gerencia dispositivos I/O
SISTEMAS OPERACIONAIS
Multinível = camadas de abstração
JCL = scripts para jobs batch em mainframe
GUI = interface gráfica (WIMP)
System Call = programa → kernel
Monotarefa = 1 processo / Multitarefa = N
Microkernel = kernel mínimo, resto fora
Thread = sub-fluxo de execução
MEMÓRIA
MMU = virtual → físico
Swap Out = RAM → Disco
Swap In = Disco → RAM
Thrashing = swap excessivo = lentidão
FIFO = menor espera na chegada
SJF = menor job primeiro, starvation
REDES
Simplex = 1 direção só
Half-duplex = 2 direções, não simultâneo
Full-duplex = 2 direções simultâneas
Hub = camada 1, repete tudo
Switch = camada 2, usa MAC
Roteador = camada 3, usa IP
TCP = confiável / UDP = rápido
